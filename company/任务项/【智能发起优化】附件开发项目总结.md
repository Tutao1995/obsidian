- PC：`/fileUpload.do?method=processUploadForH5`

- 移动端：`/seeyon/rest/attachment?token=&option.n_a_s=1&firstSave=true&sortNum=0`
- 移动端不读取 MD5 配置、不计算 SparkMD5、不查询续传起点、不分片。
- PC/移动端均使用 XHR 获取上传进度。

## 项目疑难点

### 1. V5 老附件协议与新 Schema 组件体系适配

V5 附件上传依赖老接口协议，PC 端需要兼容 `/fileUpload.do?method=processUploadForH5`、分片、续传起点、MD5 配置、密级参数和历史回调；而新组件体系要求通过 Schema、`v-model`、事件和 mixed-card action formatter 统一工作。难点在于不能只做 UI 包装，必须把老协议拆成可复用的 `core/uploadService.ts` 和 `hooks/useV5AttachmentUpload.ts`，同时保证上传成功后才写入表单值，避免上传中或失败文件污染业务数据。

### 2. PC 与移动端上传协议差异明显

PC 端沿用 V5 分片上传和续传能力，移动端则走 `/rest/attachment` 单文件上传，不能使用 SparkMD5、不能查询续传起点，也不走分片。实现时抽象了统一上传状态机，但在协议层按平台分流，既复用了校验、临时态、进度、失败态和取消逻辑，又避免把 PC 专属协议强加给移动端。

### 3. A2UI 回填与 CAP4 提交字段口径不一致

后端文档中 `fileId` 实际对应 V5 `fileUrl`，A2UI 回填又包含 `{ value, showValue, attachmentInfos }`，如果直接使用 `id` 或对象数组提交，会出现下载失败、预览失败或 CAP4 后端无法识别附件的问题。最终统一为：展示和下载以 `attachmentInfos` 的完整附件对象为准，提交 `value` 固定使用 `fileUrl` 英文逗号拼接，忽略历史 `attachmentValueMode` 干扰。

### 4. 上传失败和上传中状态不能污染表单值

附件上传有多种中间态：上传中、取消、接口失败、本地校验失败。难点在于 UI 要能展示这些状态，但表单值只能包含真正上传成功的附件。实现中将真实 `value` 与内部 `uploadingFiles` 分离，`displayValue = value + 临时项`，上传失败只保留在组件内部供用户查看和删除，不触发 `update:value/change`。

### 5. 表单提交时的上传中拦截需要跨工程解耦

附件组件在 `comi-schema-cards`，宿主发送消息和 mixed-card 提交在 `ai-assistant-web`。如果组件直接依赖宿主 store，会造成库和业务耦合。最终通过 `comiEventBus` 广播上传状态，宿主维护 `activeUploadIds` 集合，并在发送消息、mixed-card 按钮、schema form 按钮等关键入口同步 guard，解决了多附件、多卡片并发上传时的状态漂移问题。

### 6. 只读/预览态不能降级成普通文本

Schema 表单默认只读分支容易把复杂值转成字符串，附件数组会显示成 `[object Object]`。本次通过复杂值组件直渲染机制，让附件在 readonly/display 下仍走真实组件渲染，从而保留文件名、图标、大小、下载和预览样式。

### 7. 校验规则多来源且优先级复杂

附件限制同时来自后端 `attachmentConfig`、旧字段 `accept/extensions/fileExtensions/maxSize/filesNum`、V5 系统禁用后缀配置。实现中统一通过 `normalizeAttachmentUploadOptions()` 归一化，明确后端 `attachmentConfig` 优先；`include: []` 表示不启用白名单，不回退旧字段；禁用后缀仍额外生效。后续又根据业务要求移除 `minCount` 前端拦截，只保留最大数量校验。

### 8. 国际化需要兼顾可读性和运行时切换

直接使用 `t('attachment.xxx')` 虽能国际化，但调用点看不出中文含义，维护成本高。最终在现有 i18n 基础上新增 `tWithDefault(key, defaultMessage, params?)`，运行时仍走语言包，key 缺失时 fallback 默认文案，同时让代码调用处保留中文业务语义，兼顾国际化和可读性。

## 项目亮点

### 1. 完成 V5 附件能力组件化沉淀

将 V5 老附件上传协议封装为可复用 Schema 自定义组件 `comi-v5-attachment`，支持 PC/移动端上传、下载、删除、进度、失败态、只读回显、A2UI 回填和 mixed-card 提交，降低后续智能发起场景接入附件能力的成本。

### 2. 设计了成功值与临时态分离的上传状态模型

通过 `value` 和 `uploadingFiles` 分离，保证表单值只包含上传成功的附件，上传中和失败态只用于 UI 展示，避免失败文件进入业务提交数据。该模型提升了表单数据一致性，也让取消、失败、重试和关闭弹层等交互更可控。

### 3. 兼容 PC 老协议和移动端新接口

在同一组件中同时支持 PC 分片上传和移动端 REST 上传，抽象统一的 hook 状态机，协议差异下沉到 service 层，既保留 V5 老能力，又满足移动端轻量上传体验。

### 4. 打通 A2UI、Schema、mixed-card、CAP4 提交流程

从 A2UI 字段识别、Schema 组件映射、附件对象回填、前端展示，到 mixed-card action formatter 提交 `fileUrl`，形成完整链路，解决了附件字段从智能生成到后端提交的端到端闭环问题。

### 5. 通过事件总线实现跨工程上传中拦截

组件库不直接依赖宿主应用状态，而是通过事件总线同步上传状态和 guard 查询。宿主以 `uploadId` active set 管理并发任务，避免 boolean 状态在多文件、多组件场景下失真，提升了跨工程集成的稳定性。

### 6. 细化 PC/移动端体验一致性

PC 端实现输入框式编辑态、无边框预览态、hover 操作蒙层、5 行滚动和上限置灰；移动端实现 Vant popup、空态 SVG、上传中关闭保护、失败临时项清理和只读态对齐，保证多端体验符合 UE 要求。

### 7. 建立附件国际化方案

新增 `attachment.*` 文案命名空间和 `tWithDefault` 封装，覆盖上传、失败、校验、配置元数据等运行时中文，支持简体、繁体、英文，同时保留默认中文含义，提升后续维护和多语言扩展效率。

### 8. 文档和测试同步沉淀

同步更新附件 README，明确目录结构、协议差异、校验优先级、移动端行为、国际化和 `minCount` 不校验等决策；补充和调整相关单测，保证关键工具函数和 minCount 行为有测试覆盖。
