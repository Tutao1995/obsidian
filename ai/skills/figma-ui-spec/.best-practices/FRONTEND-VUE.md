# Vue 编码最佳实践

编写/审查 Vue 代码前阅读此文档。默认 Vue 2，无特殊说明按此版本编码。

任何 Vue 项目编码都必须强制遵守以下规范，违反将导致代码审查不通过。

## 核心原则

| 原则   | 说明                           |
|------|------------------------------|
| 最佳实践 | 编写符合行业最佳实践的高质量代码            |
| 风格匹配 | 确保编码风格与现有项目保持高度统一            |
| 可持续性 | 严禁使用权宜之计（Workaround），对技术债零容忍 |

## 基础开发规范

- 事件绑定必须在组件销毁前移除
- `setTimeout`/`setInterval`/`requestAnimationFrame` 等任务必须显式调用对应清除方法
- 组件内创建的外部组件实例必须在销毁时释放
- 禁止死循环、无终止条件的迭代/递归
- 禁止扩展内置对象/三方框架原型（包括 Vue），统一使用 import 导入
- 非公共组件、三方组件调用逻辑不能用 `try...catch` 隐藏业务代码错误；必须使用时必须 `console` 输出原始错误

## Vue 开发规范

- `<template>` 保持结构清晰，复杂逻辑必须外置到 `computed` 或 `methods` 中封装
- `data()` 只定义与页面状态有关的属性，其余属性在 `created()` 中初始化，减少非必要响应式属性
- `computed` 属性禁止包含修改 `data()` 属性的语句，也不能调用修改 data 的方法
- 严格遵循单向数据流：禁止 `.sync` 修饰符、禁止直接操作父组件传递的引用对象/数组
- 功能组件确保无副作用：只接受 `props` 传入参数，只通过 `$emit` 对外开放事件

## 通用规范

- **禁止 setTimeout**: setTimeout 绝大多数时候是时序问题的 Workaround，避开了根本问题
- **国际化(i18n)**: 禁止硬编码显示文本，必须遵循当前模块的国际化规范，参考已有实现，必须在 `export_to_js.xml` 中导出前端才能使用

## 代码风格

- 避免重复代码逻辑，尽可能使用已有组件和工具函数，新增重复代码提取为工具函数
- AI 产出的项目架构设计、计划等文档统一放入项目根目录下 `.ai` 目录
- 语义化命名优先：见名知意，禁止拼音、无意义缩写（如 a、b、temp、data），2~4 个单词最佳，最长不超过 6 个单词

## 测试策略

**前端单元测试为可选项。** 编码完成后询问用户："是否需要为前端代码编写单元测试？"

- **用户选择是**：编写组件/函数的单元测试
- **用户选择否**：跳过测试步骤

## 风险评估（Risks）

涉及以下场景时，spec 或 coding 阶段**必须**生成风险评估表：

| 触发条件 | 说明 |
|---------|------|
| 接口变更 | API 签名变更、返回值结构调整 |
| 状态管理 | Vuex/Pinia store 结构变更，影响多组件 |
| 路由变更 | 路由结构调整、权限路由变更 |

**风险表格式**：

| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|---------|
| {风险描述} | 高/中/低 | 高/中/低 | {具体缓解方案} |

简单 UI 调整 / 样式变更 → 跳过。

## 发布策略（Rollout）

涉及以下场景时，**必须**生成发布策略：

| 触发条件 | 说明 |
|---------|------|
| 接口变更 | 与后端接口的契约变更，需要协调发布顺序 |
| 全局组件变更 | 公共组件修改影响多模块 |

**发布策略格式**：

- **发布方式**：{一次性/灰度/分批}
- **回滚条件**：{触发回滚的指标或条件}
- **兼容性**（如适用）：{新旧版本并存策略}

纯内部页面 / 开发环境变更 → 跳过。
